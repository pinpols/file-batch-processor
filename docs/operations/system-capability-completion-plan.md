# 批量调度系统补齐实施计划

日期：2026-03-15  
目标：基于当前系统现状，按阶段补齐“成熟 Java 批量调度系统”所需的核心能力，最终形成一套可运行、可追踪、可恢复、可审计、可运维的生产级批量平台。  
配套分析文档见：[批量调度系统能力与缺口分析](./system-capability-gap-analysis.md)

## 1. 最终目标

最终补齐后的系统，至少要达到下面这些结果：

- 任务调度、批处理执行、文件接收、文件分发都纳入统一业务模型
- 文件不再只是目录中的文件，而是受管资产
- 任务和文件都有清晰状态机，并支持合法迁移与恢复
- 重试、补偿、补跑、回放都走统一入口
- 运维可通过控制台查询、补跑、重试、审计
- 告警、指标、归档、清理形成闭环
- 新旧数据结构可平滑迁移，不做一次性大爆改

## 2. 实施原则

### 2.1 渐进改造

- 不推翻现有 Quartz / Spring Batch / TaskConfig 三层主链路
- 新模型先旁路建设，再逐步切流
- 先双写、再读切换、最后退役旧路径

### 2.2 先模型、后流程、再运维

优先顺序固定为：

1. 统一数据模型
2. 统一状态流转
3. 统一补偿与恢复
4. 运维与审计闭环

### 2.3 每阶段都必须有 DoD

每个阶段都必须交付：

- Flyway 迁移
- Entity / Repository / Service
- 最小 API 或最小调度接入点
- 自动化测试
- 文档更新

### 2.4 不允许无计划直改线上主路径

所有大改都必须具备：

- 特性开关
- 回滚方案
- 数据迁移脚本
- 兼容窗口说明

## 4. 各阶段详细计划

## 阶段 0：基线冻结

### 目标

在动核心数据模型前，先把当前系统的技术基线固定住，避免“边修边漂移”。

### 需要做的事

- 固定当前主链路：
  - Quartz JDBC 调度
  - Spring Batch 执行
  - DB 配置源任务定义
- 明确新旧模型切流策略：
  - 先建新表
  - 再双写
  - 再读切换
  - 最后退役旧表
- 增加实施约束文档：
  - 哪些旧表短期保留
  - 哪些接口未来要迁移
  - 哪些表是新主模型

### 交付物

- 实施边界文档
- 数据迁移原则
- 特性开关命名规范
- 回滚原则说明

### 验收标准

- 后续每个阶段都知道“往哪改”和“怎么回退”
- 不再出现结构性重叠建设

## 阶段 1：统一领域模型

详细设计文档：

- [阶段 1 详细设计：统一领域模型](../architecture/stage1-domain-model-design.md)

### 目标

先统一概念，不直接改业务主路径。

### 需要定义的核心对象

- 文件资产
- 文件处理日志
- 文件分发记录
- 业务任务实例
- 任务步骤实例
- 任务执行日志
- 文件状态枚举
- 任务状态枚举

### 建议输出

- 领域模型文档
- 状态流转图
- 事件清单
- 表结构草案

### 验收标准

- 每个现有表都能映射到新模型中的某个角色
- 不再出现“同一个概念在多个表里各存一半”的歧义

## 阶段 2：文件资产主表

表结构与 Flyway 草案：

- [V1_27 文件资产模型草案](../architecture/sql-drafts/V1_27__file_asset_model_draft.sql)

### 目标

把文件从“临时业务对象”升级为“统一受管资产”。

### 新增表

- `file_record`
- `file_process_log`
- `file_dispatch_record`

### `file_record` 建议字段

- `id`
- `file_no`
- `source_system`
- `biz_type`
- `original_name`
- `stored_name`
- `stored_path`
- `file_size`
- `file_hash`
- `file_ext`
- `charset`
- `biz_date`
- `batch_no`
- `status`
- `version`
- `arrived_time`
- `processed_time`
- `archived_time`
- `deletable`
- `deleted_flag`

### 需要改的代码

- 新增文件领域 service
- `FileReceptionService` 接入 `file_record`
- `FileDistributionService` 接入 `file_record`
- 导出链路产生出站文件时也落 `file_record`

### 切流策略

- 第一阶段只做双写，不切旧表读路径
- `file_reception_queue` 和 `file_distribution_task` 先继续保留

### 验收标准

- 任意入站、出站文件都能在 `file_record` 查到
- 旧链路不受影响
- 新增集成测试覆盖接收、导出、分发写入 `file_record`

## 阶段 3：文件状态机

### 目标

统一文件状态流转，收敛当前散落的状态写法。

### 建议状态

- `UPLOADING`
- `ARRIVED`
- `READY`
- `PROCESSING`
- `PROCESSED`
- `FAILED`
- `DISPATCHING`
- `DISPATCHED`
- `ARCHIVED`

### 需要做的事

- 新增 `FileLifecycleService`
- 定义合法迁移矩阵
- 文件状态变化统一写 `file_process_log`
- 明确终态和可恢复态

### 需要改的代码

- `FileReceptionService`
- `FileDistributionService`
- `FileExportService`
- 与文件有关的 Tasklet/Job

### 验收标准

- 文件状态不再直接散写字符串
- 非法状态迁移会被拦截
- 每次状态变化都有日志

## 阶段 4：半文件防护与文件幂等

### 目标

优先解决文件入口最容易出事故的部分。

### 需要做的事

- 支持 `.part -> final rename` 协议
- 或支持 `.done` 标志文件
- 增加“文件大小稳定检测”兜底
- 文件判重从“按文件名”升级到“按内容 + 业务维度”

### 判重建议

- `source_system + file_hash + biz_date`
- 必要时补 `batch_no`
- 同名不同内容单独标记冲突

### 需要改的代码

- `FileReceptionService`
- 文件扫描任务
- 接收任务参数规范

### 验收标准

- 半文件不会进入处理链路
- 重复投递不会重复处理
- 冲突文件会被识别并留痕

## 阶段 5：任务实例服务

### 目标

建立统一业务任务实例层，避免把 Spring Batch / Quartz 元数据直接当业务任务中心。

### 新增表

- `job_instance`
- `job_step_instance`
- `job_execution_log`

### `job_instance` 最少要记录

- `task_id`
- `job_name`
- `trigger_source`
- `operator`
- `biz_date`
- `batch_no`
- `status`
- `rerun_flag`
- `retry_flag`
- `related_file_id`
- `start_time`
- `end_time`
- `result_summary`

### 需要做的事

- `LaunchExecutor` 启动任务时创建业务任务实例
- Job/Step 监听器同步更新实例状态
- 任务实例关联文件资产

### 验收标准

- 任意一次任务都能从业务维度完整查询
- Quartz 执行记录、Batch 执行记录、业务任务实例能串起来

## 阶段 6：重试补偿统一化

### 目标

把当前分散的补偿路径统一收口。

### 当前已有能力

- `BatchRecoveryService`
- `DlqCompensationService`
- 文件分发 retry

### 需要统一成的能力

- 任务级重试
- 步骤级补偿
- 文件级重处理
- 批次级补跑
- 死信回放

### 建议新增

- `RetryCompensationService`
- `CompensationCommand`
- `CompensationRecord`

### 运维能力

- 指定 task 补跑
- 指定 biz_date 补跑
- 指定 file_id 重处理
- 指定 step 重试

### 验收标准

- 所有重试动作有统一入口和统一审计
- 失败恢复不再分散在多个 service 里各写一套

## 阶段 7：出站分发与回执闭环

### 目标

把导出和分发做成受控产品链路，而不是“生成文件后直接发”。

### 需要做的事

- 出站文件版本化
- 分发记录标准化
- 分发幂等键
- 回执状态与回执时间
- 下游未取走超时监控

### 建议补充字段

- `dispatch_channel`
- `dispatch_status`
- `ack_status`
- `ack_time`
- `version`

### 验收标准

- 任意一个导出文件都能查到“生成版本、发给谁、是否成功、是否回执”

## 阶段 8：运维台与审计台

### 目标

让运维操作尽量脱离直接查库。

### 需要增加的页面或接口

- 文件检索
- 文件处理日志查询
- 文件分发记录查询
- 文件重处理
- 批次补跑
- 失败任务补偿
- 文件下载鉴权

### 审计需要覆盖

- 谁手工触发
- 谁补跑
- 谁重发
- 谁改参数
- 谁下载文件

### 验收标准

- 常见故障排查、补跑、重传都能通过控制台完成

## 阶段 9：告警、监控、归档清理

### 目标

补齐长期稳定运行所需的运营能力。

### 需要补的告警

- 文件超时未到达
- 文件到达未处理
- 分发未回执
- 磁盘空间不足
- 目录堆积
- 文件归档失败

### 需要补的指标

- 文件接收量
- 文件失败率
- 分发成功率
- 回执延迟
- 文件处理平均耗时
- 文件积压量
- 归档量

### 需要补的归档能力

- 原始文件保留周期
- 处理中间文件保留周期
- 导出文件保留周期
- 审计文件长期保留策略
- 归档前校验是否终态、是否还有补跑依赖

### 验收标准

- 系统运行一段时间后不会靠人工清理才能维持
- 文件生命周期有明确归档和删除策略

## 阶段 10：切流与旧结构退役

### 目标

把新模型变成主路径，旧路径进入兼容或退役状态。

### 需要做的事

- 新旧表双写检查
- 回填历史数据
- 读路径切换到新模型
- 旧表只读化或兼容化
- 最终退役旧路径

### 切流顺序

1. 新表建好
2. 新旧双写
3. 运维台和查询读新表
4. 业务服务读新表
5. 停止旧写入
6. 观察稳定后退役旧逻辑

### 验收标准

- 10 个任务全链路跑通
- 文件接收、处理、导出、分发、补偿、归档都走新模型
- 旧结构不再承担主功能

## 6. 每阶段完成定义

每个阶段完成前，必须同时满足：

- 代码实现完成
- Flyway 已提交
- 自动化测试通过
- `mvn test` 不回归
- 至少一条联调或组件集成验证通过
- 文档更新完成
- 明确下一阶段输入输出

## 7. 建议的首批实施范围

如果按“先做最有价值、最不容易返工”的原则，建议先启动下面 3 个阶段：

1. 阶段 1：统一领域模型
2. 阶段 2：文件资产主表
3. 阶段 3：文件状态机

原因：

- 这三步一旦定清楚，后面的补偿、补跑、运维、告警都能挂上去
- 如果这三步不先定，后面继续加功能只会把现有模型再拉散一次
